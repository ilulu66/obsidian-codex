# Obsidian Codex 项目复盘素材包

整理时间：2026-07-02  
项目仓库：https://github.com/ilulu66/obsidian-codex  
当前版本：v0.4.0

## 这份资料是什么

这是给 Claude 写项目复盘文章用的素材包。

它记录了我们从一个真实使用痛点出发，把一个本地 Obsidian 插件一步步迭代成可发布 GitHub 项目的全过程。重点不是代码实现，而是：

- 用户需求如何被提出
- 使用中的摩擦如何暴露
- 每次反馈如何转化成产品改动
- 这个项目体现了什么 AI 协作方法
- 这套过程能写成什么样的文章

## 一句话总结

这个项目的核心不是“写了一个 Obsidian 插件”，而是把 Codex 从一个外部命令行工具，嵌入到 Obsidian 的知识工作流里，让它逐渐从“能调用”变成“能日常协作”。

## 项目起点

用户一开始的原话：

> 我现在在用 obsidian + claudian，但是没有 codex 的插件，在 obsidian 就不方便调用你，你能做一个插件吗，参考 claudian 的做法

这个需求里隐含了三个关键前提：

1. 用户已经有稳定工作流：Obsidian + Claudian。
2. 用户不是想换工具，而是想把 Codex 接进现有工作流。
3. “参考 Claudian”意味着插件不应该只是命令行包装，而应该像 Obsidian 内部助手。

## 项目演进总览

### 阶段 1：先跑通 Obsidian 内调用 Codex

目标：能在 Obsidian 里打开一个 Codex 面板，并调用本机 `codex exec`。

关键动作：

- 读取本地 `.obsidian/plugins/claudian` 的插件结构。
- 参考 Claudian 的形态：Obsidian `ItemView`、侧边面板、ribbon 图标、命令面板入口。
- 创建 `.obsidian/plugins/codex`。
- 写入 `manifest.json`、`main.js`、`styles.css`。
- 默认使用当前 vault 作为 Codex 工作目录。
- 增加命令：
  - `Codex: Open Codex view`
  - `Codex: Ask Codex about selection`
  - `Codex: Ask Codex about current note`
  - `Codex: Insert last Codex response at cursor`
  - `Codex: Stop current Codex run`

产品意义：

从“用户要切出 Obsidian 才能调用 Codex”，变成“Obsidian 内部能直接问 Codex”。

### 阶段 2：让用户知道它真的在工作

用户反馈：

> 我能看到 codex，但发了消息，它好像不工作？

实际情况：

Codex 已经在后台完成，并生成了 `last-response.md`，但是 UI 没有明确显示运行状态和结果。

改动：

- 增加运行状态提示。
- 增加“最近回复”按钮。
- 过滤 Codex 启动时的 skill-loading 噪音。
- 显示更明确的错误提示。

产品意义：

后台成功不等于用户成功。一个 AI 工具如果没有状态反馈，用户会认为它坏了。

### 阶段 3：从单窗口变成 5 个固定工作窗口

用户反馈：

> 我希望这个也有 5 个窗口，像 claudian 一样

改动：

- 增加顶部 `1 2 3 4 5` 五个固定窗口。
- 每个窗口独立保存消息记录。
- 每个窗口独立保存最近回复。
- 命令面板增加：
  - `Codex: Open Codex window 1`
  - `Codex: Open Codex window 2`
  - `Codex: Open Codex window 3`
  - `Codex: Open Codex window 4`
  - `Codex: Open Codex window 5`
- 运行中禁止切换窗口，避免结果写错窗口。

产品意义：

这个反馈说明用户需要的不是一次性问答，而是多个并行任务空间。5 个窗口让插件更像一个工作台。

### 阶段 4：打包到 GitHub，让它从个人插件变成可复用项目

用户反馈：

> 能不能帮我打包到 github，我在 claude 好像做过打包的流程，你看看

改动：

- 找到参考项目 `50-项目代码/codex-plugin-cc`、`topic-surgeon`。
- 创建独立项目目录：
  - `50-项目代码/obsidian-codex`
- 建立 GitHub 仓库：
  - https://github.com/ilulu66/obsidian-codex
- 添加：
  - `README.md`
  - `LICENSE`
  - `CHANGELOG.md`
  - `versions.json`
  - `package.json`
  - `scripts/check-manifest.mjs`
  - `scripts/package.mjs`
  - GitHub Actions CI
  - Release workflow
- 创建 release：
  - v0.2.0

特别注意：

- 没有上传本地运行数据：
  - `data.json`
  - `last-response.md`
  - `window-*-last-response.md`
- 移除了 README 里的本机绝对路径。

产品意义：

项目从“本地能用”变成“别人能安装、能理解、能复用”。

### 阶段 5：把说明改成中文友好版

用户反馈：

> 我感觉你能不能写个中文的说明能友好一点

改动：

- 将 README 从英文工程说明改成中文用户说明。
- 结构改成：
  - 这个插件是什么
  - 它能做什么
  - 安装前准备
  - 安装插件
  - 第一次使用
  - 常用命令
  - 如果它没有反应
  - 默认安全设置
  - 本地数据说明
  - 开发者说明
- Release 页面也改成中文安装说明。

产品意义：

文档不是给开发者自嗨，而是降低使用门槛。中文 README 让项目更像一个真正面向用户的工具。

### 阶段 6：让对话区像正常聊天，而不是终端日志

用户反馈：

> 这个插件的对话区内容没办法部分复制，也没办法选中，我觉得有问题。另外前面会跑很多代码语言，我觉得我好奇思考过程，但是代码不用给我看，思考过程和结果

改动：

- CSS 明确允许消息区 `user-select: text`。
- 消息正文可局部选中、可部分复制。
- 运行时不再显示命令行参数和终端日志。
- 默认提示词改成要求输出：
  - `思考过程（简要）`
  - `结果`
- 默认不输出大段代码和运行日志，除非用户明确要求。
- 发布 v0.2.1。

产品意义：

用户要的是协作结果，不是机器执行噪音。AI 插件嵌进 Obsidian 后，应该服务阅读、复制、整理，而不是暴露终端过程。

### 阶段 7：输入体验补齐最后一厘米

用户反馈：

> 另外就是回车没办法发送内容，还需要点一下发送按钮

改动：

- `Enter` 发送。
- `Shift + Enter` 换行。
- 兼容中文输入法组合状态，选词时按回车不会误发送。
- 发布 v0.2.2。

产品意义：

高频工具的体验常常死在最后一厘米。每次都要点按钮，会让用户不愿意频繁使用。

### 阶段 8：复盘产品逻辑，确定下一轮优先级

用户请求：

> 你帮我复盘梳理我们整个使用迭代逻辑，我给的启发有哪些？

复盘结论：

这次迭代不是“堆功能”，而是沿着一个清晰路线推进：

```text
能跑通 -> 看得见 -> 像原工具 -> 能分发 -> 看得懂 -> 用得顺 -> 更像人类协作
```

用户给出的启发包括：

- 参考 Claudian：不要凭空设计，要站在用户已有心智上做。
- 在 Obsidian 里调用：AI 能力要嵌入现有工作流。
- 发了消息像没工作：后台成功不等于用户成功，状态反馈很重要。
- 要 5 个窗口：真实工作不是单线程问答，而是多任务并行。
- 打包到 GitHub：工具要从个人脚本走向可复用产品。
- 中文说明友好一点：文档要降低门槛。
- 不能部分复制：内容消费体验是核心。
- 不想看代码日志：用户要的是思考过程和结果，不是机器噪音。
- 回车发送：交互习惯决定工具能否高频使用。

### 阶段 9：确定下一轮优化方向

用户问：

> 你认为还有什么优化思路呢

提出的方向：

1. 真正的多轮上下文
2. 结果结构更稳定
3. Markdown 渲染
4. 一键插入方式更细
5. 当前笔记上下文更可控
6. 更好的运行状态
7. 安装/更新体验
8. 窗口命名

用户选择：

> 1568 可不可以先做

对应任务：

- 1：真实多轮上下文
- 5：上下文更可控
- 6：更好的运行状态
- 8：窗口命名

### 阶段 10：v0.3.0，把插件升级成更像工作台

改动：

- 每个窗口保存自己的 Codex `sessionId`。
- 第一次提问创建新 session。
- 后续提问自动使用 `codex exec resume <sessionId>`。
- 清空窗口时同时清空这个窗口的 session。
- 输入区新增上下文下拉：
  - 不带
  - 选中
  - 小节
  - 整篇
  - 反链
  - 同标签
- 反链和同标签各最多带 5 篇，避免上下文过大。
- 支持窗口命名：
  - 点“命名”
  - 双击窗口标签
- 状态更清晰：
  - 正在准备上下文
  - 正在创建 Codex 会话
  - 正在续接 Codex 会话
  - 完成
  - 错误
- 发布 v0.3.0。

验证：

- `node --check` 通过。
- `npm test` 通过。
- `npm run package` 生成 zip。
- 真实 CLI 验证：先创建 session，再用 session id 续接，返回正常。
- GitHub CI 成功。
- Release workflow 成功。

### 阶段 11：v0.4.0，让长期运行更可控

用户反馈：

> 更好的运行状态，比如显示：正在读取上下文、正在调用 Codex、已运行多少秒、可停止、失败原因。这个状态我希望是能看到的。现在这个插件我发现没办法增加窗口，好像被重命名挡住窗口。另外就是好像没有在桌面端看到这个对话，我有办法恢复吗？如果我关掉了窗口。另外就是输入框现在不支持富文本，没办法插入图片。

改动：

- 增加固定可见的运行状态面板：
  - 正在读取上下文
  - 正在调用 Codex
  - 已运行秒数
  - 停止按钮
  - 失败原因
- 顶部增加 `+` 按钮，允许继续新增窗口，最多 20 个。
- 窗口标签栏支持横向滚动，避免长窗口名挡住后面的按钮。
- 增加 `Codex: Restore Codex panel / 恢复 Codex 面板` 命令。
- 支持通过 `图片` 按钮、粘贴、拖拽添加图片。
- 图片会保存到本地插件附件目录，并通过 Codex CLI 的 `-i` 参数传入。
- README 明确说明：Obsidian 插件会保存本地对话，但这些对话不一定出现在 Codex 桌面端侧边栏。

产品意义：

这一轮不是“加功能炫技”，而是补齐真实使用时最需要的可控性：知道它在做什么、跑了多久、怎么停、失败在哪里、窗口关了怎么回来、图片怎么进入上下文。

## 用户启发清单

### 1. “参考 Claudian”

这句话非常关键。它不是说复制 Claudian，而是给出了用户心智锚点：

- 用户熟悉侧边聊天面板。
- 用户熟悉多个窗口。
- 用户希望 AI 直接工作在 vault 里。

启发：

产品设计不应该从功能列表出发，而应该从用户已经形成的操作习惯出发。

### 2. “发了消息好像不工作”

这句话暴露的是可感知性问题。

事实是后台已经跑完，但用户不知道。

启发：

AI 工具必须管理等待体验。等待不可怕，不知道它在不在工作才可怕。

### 3. “也有 5 个窗口”

这句话把插件从单次问答工具推向多任务工作台。

启发：

用户在知识工作里不是只有一个问题，而是同时有多个项目、文章、想法、调试任务。窗口不是 UI 装饰，而是任务分区。

### 4. “打包到 GitHub”

这句话推动项目从本地解决方案变成可分发工具。

启发：

一个工具能不能被复用，取决于它有没有清晰的项目结构、文档、release 和安装路径。

### 5. “中文说明友好一点”

这句话说明工程完成不等于产品完成。

启发：

README 是产品的一部分。尤其是给非纯开发者使用的工具，文档必须先回答“我怎么用”，而不是先展示“我怎么实现”。

### 6. “没办法部分复制”

这是内容工具的基本体验。

启发：

AI 输出不是看完就结束，用户会摘取、复制、整理、二次编辑。可选中、可复制、可插入是知识管理场景的基础能力。

### 7. “代码不用给我看，思考过程和结果”

这句话定义了插件的输出风格。

启发：

用户不是要看机器怎么跑，而是要看人能理解的判断过程和结果。AI 工具要隐藏执行噪音，呈现协作信息。

### 8. “回车发送”

这是高频交互的手感问题。

启发：

高频工具必须符合直觉。每多一次点击，都会降低使用频率。

### 9. “1568 先做”

这句话体现了用户的产品优先级判断：

- 先让工作台真的可持续协作。
- 先让上下文更可控。
- 先让状态更明确。
- 先让窗口有名字。

启发：

下一步不是做炫技功能，而是让核心工作流更稳。

## 文章可写的核心主题

### 主题 1：AI 工具不是越强越好，而是越顺手越好

这个项目里很多关键改动都不是模型能力，而是工作流细节：

- 等待状态
- 复制体验
- Enter 发送
- 中文说明
- 窗口命名

文章可以写：真正决定 AI 工具能否被长期使用的，不是第一次惊艳，而是日常摩擦是否足够低。

### 主题 2：从“调用 AI”到“嵌入工作流”

一开始只是想在 Obsidian 里调用 Codex。后来逐渐变成：

- 在当前笔记里调用
- 带选中文本
- 带小节
- 带反链
- 带同标签笔记
- 每个窗口保留 session

文章可以写：AI 的价值不是孤立问答，而是进入已有知识系统。

### 主题 3：用户反馈如何驱动产品演进

每次反馈都不是抽象建议，而是来自真实使用：

- “好像不工作”
- “要 5 个窗口”
- “不能复制”
- “代码不用给我看”
- “回车不能发送”

文章可以写：产品感不是规划出来的，是在真实使用中被磨出来的。

### 主题 4：AI 协作里的用户角色

这次项目里，用户不是只提需求，而是在不断校正产品方向：

- 从 Claudian 心智出发
- 要 GitHub 打包
- 要中文文档
- 要更像聊天
- 要更像工作台

文章可以写：AI 时代的产品创造，不是“人提需求，AI 执行”，而是“人提供品味、场景和判断，AI 负责快速实现与验证”。

## Claude 写作注意事项

写文章时不要写成纯技术教程。重点应该是：

- 一个真实工具是怎么被磨出来的
- 用户反馈如何变成产品设计
- AI 协作中人的判断力有多重要
- Obsidian 这种个人知识库为什么适合嵌入 AI

语气建议：

- 第一人称
- 有过程感
- 不要过度技术化
- 不要堆代码
- 保留用户原话的生活感
- 重点写“为什么这样改”，不只写“改了什么”
